Error Codes

Diseño base para comunicar errores del sistema con el mismo lenguaje visual del proyecto. Esta vista sirve tanto para documentación como para inspiración de futuras pantallas de error en la aplicación.

Alineado con el proyecto Listo para docs Escalable
Resumen

La propuesta distingue errores de acceso, validación, conflicto y disponibilidad. Cada bloque mantiene tono operativo, causa resumida y siguiente acción sugerida para que el usuario no se quede sin salida.

Errores cliente4xx

Entrada, permisos o rutas.

Errores servidor5xx

Fallo interno o servicio no disponible.

TonoClaro

Mensaje breve, sin dramatismo.

AcciónGuiada

Siempre hay siguiente paso.

Catálogo principal
400
Solicitud inválida
Bad Request
La petición llegó, pero faltan campos, formato o parámetros esperados por el sistema.
  • Validar payload o query params.
  • Mostrar el campo conflictivo cuando exista contexto.
  • Registrar detalle técnico con requestId.
401
Autenticación
Unauthorized
La operación requiere sesión válida o un token vigente antes de continuar.
  • Invitar a iniciar sesión de nuevo.
  • Explicar si el token expiró o falta.
  • Preservar la intención del usuario para reintento.
403
Permisos
Forbidden
La identidad es conocida, pero no tiene permisos para completar la acción solicitada.
  • Mostrar el rol requerido cuando sea posible.
  • Evitar lenguaje ambiguo con 401.
  • Ofrecer volver al recurso permitido.
404
Ruta o recurso
Not Found
El recurso no existe, fue movido o la URL solicitada ya no está disponible.
  • Ofrecer acceso a inicio o navegación principal.
  • Permitir volver al reporte oficial o índice.
  • Registrar la ruta fallida para análisis.
409
Conflicto
Conflict
La operación no puede completarse porque el estado actual del recurso entra en conflicto con la solicitud.
  • Útil para edición concurrente o versión antigua.
  • Sugerir refrescar antes de reintentar.
  • Adjuntar versión esperada y versión actual.
422
Validación
Unprocessable Entity
La estructura de la solicitud es correcta, pero los datos no superan validaciones de negocio o integridad.
  • Resaltar reglas incumplidas.
  • Asociar cada error a un campo concreto.
  • Evitar mensajes genéricos tipo “ha fallado”.
500
Fallo interno
Internal Server Error
El sistema encontró una condición no controlada mientras intentaba completar la operación.
  • No exponer stacktrace al usuario final.
  • Mostrar un mensaje sobrio y accionable.
  • Registrar contexto para soporte y rollback.
503
Disponibilidad
Service Unavailable
El servicio está temporalmente fuera de línea, degradado o en mantenimiento.
  • Indicar si conviene reintentar más tarde.
  • Mostrar estado parcial si existe degradación.
  • Reservar esta vista para incidentes temporales.
Uso recomendado

Si más adelante lo llevas a la aplicación, la convención visual ya está decidida: código grande, estado en badge, explicación en una frase y una lista corta de acciones. Para 404 y 500, conviene además incluir CTA primario a Inicio y secundario a Volver o Reintentar.
